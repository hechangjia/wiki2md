# wiki2md v0.2 Polish Design

## Goal

Improve `wiki2md` so each converted person article is a more complete, higher-quality corpus unit for daily use.

This phase focuses on three content-quality upgrades:

1. extract and preserve person-page infobox data
2. improve image metadata quality, especially captions and alt text
3. make body structure more stable so paragraphs, headings, and lists survive normalization more reliably

This is the first sub-project of a broader `v0.2 polish` effort. Follow-on work for batch workflow and open-source packaging is intentionally deferred to later specs.

## Why This Phase First

The user priority is article quality over tooling polish.

For corpus-building, the highest-value improvement is not a more elaborate CLI. It is making each saved article materially better:

- more complete person facts
- better preserved visual context
- cleaner, more stable Markdown
- stronger machine-readable sidecars

If this layer is weak, batch processing only scales weak artifacts faster.

## Scope

### In Scope

- Person-page infobox extraction from English-first, Chinese-compatible Wikipedia pages
- New `infobox.json` sidecar for structured infobox output
- A readable infobox section rendered into `article.md`
- Better caption and alt fallback behavior for kept images
- More stable extraction of prose-oriented body blocks
- Light metadata extensions needed to support the richer artifact contract

### Out of Scope

- Large batch-workflow redesign
- New sync/update workflows
- Full table preservation
- Timeline or works-list parsing as first-class structures
- Generic support for non-person page types
- Raw `source.html` or fetch-trace artifact storage
- Full open-source polish such as CI, license selection, release workflow, and repository hygiene automation

## Output Contract

The output directory for a converted person page should now include one additional sidecar:

```text
output/
  people/
    geoffrey-hinton/
      article.md
      meta.json
      references.json
      infobox.json
      assets/
```

### article.md

`article.md` remains the primary reading artifact.

It should keep:

- existing frontmatter
- title heading
- clean lead paragraphs
- body headings, paragraphs, lists, and kept images
- compact reference summary

It should gain a new human-readable profile section near the top:

```md
# Geoffrey Hinton

![Geoffrey Hinton](./assets/001-infobox.jpg)

## Profile

- Born: 6 December 1947 Wimbledon, London, England
- Citizenship: United Kingdom, Canada
- Known for: backpropagation, Boltzmann machine, deep learning

Geoffrey Hinton is a British-Canadian computer scientist ...
```

Rendering rules:

- If an infobox image exists and is kept locally, render it directly under the title.
- Render `## Profile` before the lead summary.
- Render one bullet per infobox field in source order.
- Keep the Markdown profile section readable and compact; do not dump raw HTML or template syntax.
- Preserve the existing clean-first prose rule: inline Wikipedia citation markers should not appear in ordinary paragraph or list prose.

### infobox.json

`infobox.json` is the machine-readable sidecar for infobox content.

Recommended shape:

```json
{
  "title": "Geoffrey Hinton",
  "image": {
    "title": "File:Geoffrey_Hinton_at_UBC.jpg",
    "path": "assets/001-infobox.jpg",
    "alt": "Geoffrey Hinton",
    "caption": "Hinton at the University of British Columbia in 2015"
  },
  "fields": [
    {
      "label": "Born",
      "text": "6 December 1947 Wimbledon, London, England",
      "links": [
        {
          "text": "Wimbledon",
          "href": "https://en.wikipedia.org/wiki/Wimbledon,_London"
        }
      ]
    }
  ]
}
```

Rules:

- Preserve field order from the source infobox.
- Use a field array rather than a JSON object so duplicate labels and ordering remain stable.
- `label` is required.
- `text` is the cleaned human-readable value.
- `links` is required and should be an empty array when a field has no links.
- `image` may be `null` if the page has no usable infobox image.

### meta.json

`meta.json` should remain the canonical artifact metadata file.

This phase should extend `cleanup_stats` with infobox-related counters:

- `cleanup_stats.infobox_fields`
- `cleanup_stats.has_infobox`

The goal is observability, not a second copy of `infobox.json`.

## Infobox Extraction Rules

Infobox extraction should be conservative and deterministic.

Keep:

- visible label/value rows
- simple text values
- wiki or external links embedded in values
- the primary infobox image when it maps to a kept local asset

Drop:

- empty rows
- purely decorative rows
- rows whose content is only hidden metadata or edit controls
- pronunciation/audio icons
- template leftovers with no readable article value

Normalization rules:

- Collapse internal formatting to readable text
- Preserve comma-separated or list-like value meaning in plain text
- Avoid copying raw HTML, CSS classes, or template syntax into output
- Prefer readable value text over exact HTML fidelity

## Image Quality Rules

This phase should improve image metadata quality without changing the local-asset strategy.

### Caption Priority

For any kept image, caption extraction should prefer:

1. explicit caption node such as `.infobox-caption` or `figcaption`
2. meaningful `alt` text
3. cleaned page/image title fallback when no better caption exists

### Alt Priority

Markdown image alt text should prefer:

1. explicit image `alt`
2. caption text when `alt` is empty
3. article title for the infobox image
4. a deterministic fallback such as `"Image"` only as a last resort

This avoids `![](...)` output whenever a better descriptive value is available.

## Body Stability Rules

This phase does not introduce new major block types. It strengthens the existing paragraph/list/heading/image pipeline.

Expected improvements:

- preserve heading order more reliably
- keep lead paragraphs intact
- avoid dropping valid prose because of inline citation cleanup
- reduce noise leakage from nested or citation-only nodes
- keep top-level list items readable without obvious template residue

Non-goals for body structure in this phase:

- nested list rendering
- general-purpose table support
- timeline-specific parsing
- works/awards/discography blocks as custom structures

## Architecture Changes

The existing pipeline remains the backbone, but the internal document model should gain an explicit infobox boundary.

### document.py

Add dedicated infobox structures:

- `InfoboxField`
- `InfoboxImage`
- `InfoboxData`

`Document` should gain an optional `infobox` field rather than treating all infobox content as ordinary body blocks.

### normalize.py

Responsibilities added in this phase:

- parse infobox table rows into `InfoboxData`
- extract link-aware infobox fields
- separate the infobox image from ordinary body figures
- improve prose cleanup without weakening genuine content
- improve caption and alt fallback behavior for kept images

### render_markdown.py

Responsibilities added in this phase:

- render an infobox image directly below the page title when available
- render `## Profile` from normalized infobox fields
- keep the rest of the article structure unchanged unless required for ordering stability

### writer.py

Responsibilities added in this phase:

- continue writing `article.md`, `meta.json`, `references.json`, and assets
- add `infobox.json`

### service.py

Responsibilities added in this phase:

- pass normalized infobox data through rendering and writing
- expose infobox-related cleanup stats if included in metadata

## CLI Surface

This phase should avoid widening the CLI surface unnecessarily.

Command set should remain:

- `wiki2md convert`
- `wiki2md inspect`
- `wiki2md batch`

No new top-level command is required for infobox support.

If any CLI change is needed in implementation, it should be limited to clearer output or metadata display, not a command redesign.

## Testing

Required coverage for this phase:

- normalization tests for English infobox extraction
- normalization tests for Chinese-compatible infobox extraction
- renderer tests proving `article.md` gains a readable `## Profile` section
- writer/service tests proving `infobox.json` is emitted
- tests for image caption and alt fallback behavior
- regression tests proving existing clean-prose behavior still holds
- live smoke verification on at least one English person page and one Chinese person page with infobox content

Useful live smoke candidates:

- `https://en.wikipedia.org/wiki/Geoffrey_Hinton`
- `https://en.wikipedia.org/wiki/Fei-Fei_Li`
- `https://zh.wikipedia.org/wiki/姚明`

## Risks And Tradeoffs

### Infobox Heterogeneity

Wikipedia infoboxes are not fully uniform. A deterministic field-array model is more robust than trying to force all person pages into a rigid schema.

Tradeoff:

- less semantic normalization
- better stability across real pages

### Readability vs Completeness

Rendering the full infobox into Markdown increases article length, but this is acceptable because person-page infoboxes usually contain high-value facts.

The JSON sidecar still carries the more structured representation for downstream tools.

### Incremental Scope Control

Batch UX and repository polish are important, but they are separate value streams.

This phase deliberately chooses corpus quality first so later CLI or batch improvements scale better artifacts.

## Follow-On Work

This phase should be followed by separate specs for:

1. batch and output-directory workflow polish
2. reference and evidence-sidecar refinement beyond the current shape
3. repository/open-source polish such as `.gitignore`, CI, license, and release automation

## Recommendation

Treat this as `v0.2 phase 1`.

Success means:

- person pages produce richer, more complete Markdown
- infobox facts are preserved in both human-readable and machine-readable forms
- image descriptions improve
- body extraction remains clean and stable

Only after that should the project spend serious effort on batch workflow and public-repo polish.
