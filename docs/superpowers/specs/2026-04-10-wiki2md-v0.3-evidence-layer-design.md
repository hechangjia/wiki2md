# wiki2md v0.3 Evidence Layer Design

Date: 2026-04-10
Status: Approved design

## Context

`wiki2md` already produces a clean article bundle for Wikipedia person pages:

- `article.md`
- `meta.json`
- `references.json`
- `infobox.json`
- `assets/`

The current output is strong for reading and corpus storage, but it is still weak on one important dimension for AI / RAG use cases: evidence traceability. The article body is intentionally clean, yet there is no dedicated artifact that answers:

- which references are explicitly cited within each section
- which primary source URLs are most relevant to a section
- which section-level evidence can be consumed by downstream indexing pipelines

## Problem Statement

The project currently preserves references at the page level only. `references.json` is useful as a raw sidecar, but it does not provide a section-level mapping from article structure to evidence.

This creates two gaps:

1. downstream systems cannot reliably attach section-level provenance without reparsing the article
2. human review of section-by-section sourcing is harder than it needs to be

## Goals

- Add a section-level evidence layer without making `article.md` noisy again
- Keep `article.md` clean and footnote-free
- Preserve explicit citation mappings when they are present in the source HTML
- Emit one machine-first artifact and one human-readable artifact
- Keep the first version scoped to section-level evidence only

## Non-Goals

- Do not reintroduce Markdown footnotes into `article.md`
- Do not guess provenance using text similarity
- Do not implement paragraph-level evidence mapping in this phase
- Do not introduce source quality scoring or ranking in this phase
- Do not expand this phase into a generic chunking system

## Output Contract

This phase adds two new output files to each article bundle:

```text
output/
  person/
    default/
      elon-musk/
        article.md
        meta.json
        references.json
        infobox.json
        section_evidence.json
        sources.md
        assets/
```

### `section_evidence.json`

Machine-first artifact. It should be stable, explicit, and easy to consume downstream.

Each section entry should include at least:

- `section_id`
- `heading`
- `level`
- `paragraph_count`
- `reference_ids`
- `reference_count`
- `primary_urls`

Recommended shape:

```json
{
  "title": "Andrej Karpathy",
  "sections": [
    {
      "section_id": "career",
      "heading": "Career",
      "level": 2,
      "paragraph_count": 3,
      "reference_ids": ["cite_note-12", "cite_note-14"],
      "reference_count": 2,
      "primary_urls": [
        "https://openai.com/...",
        "https://www.nytimes.com/..."
      ]
    }
  ]
}
```

### `sources.md`

Human-readable companion artifact. It should stay compact and readable.

Structure:

- grouped by article section
- no global source table in v0.3
- each section lists only the sources explicitly mapped to it

Each source entry should show:

- a short text label
- `primary_url` when available
- lightweight link-kind summary when helpful

## Core Principle

Section-level evidence must come from explicit citation traces, not post-hoc guessing.

The system should preserve citation anchors internally during normalization, map them to `ReferenceEntry` records, and then emit a section-level aggregation layer from those explicit links.

If a section has no explicit citation mapping, it is allowed to produce an empty source list.

## Recommended Approach

Use a lightweight internal citation-preservation layer during normalization.

The recommended flow is:

1. parse article HTML
2. extract explicit citation markers before prose cleanup strips them from visible text
3. attach citation ids to the section being normalized
4. aggregate section-level references after normalization
5. write `section_evidence.json`
6. render `sources.md` from the section evidence artifact

This keeps `article.md` clean while still preserving provenance in sidecars.

## Architecture

### 1. Extend Internal Document Structure

The current document model is block-based and optimized for clean rendering. v0.3 needs enough additional structure to know:

- which section each block belongs to
- which explicit citation ids were observed within that section

This does not require a full rewrite of rendering. It requires a section-aware internal representation or a section index layered on top of the current block stream.

### 2. Preserve Citation Traces Internally

Before `_clean_prose_text()` removes reference superscripts from the visible prose, normalization should capture the citation ids embedded in a node, for example:

- fragment ids that contain `cite_note`
- citation-ref style targets that resolve to reference list entries

These ids are internal-only metadata. They are not rendered back into `article.md`.

### 3. Build Section-Level Evidence

After normalization, the system should aggregate:

- section metadata
- unique citation ids seen in that section
- resolved `ReferenceEntry` objects from `references.json`
- de-duplicated `primary_urls`

This becomes the source of truth for `section_evidence.json`.

### 4. Render Human-Readable Sources From Evidence

`sources.md` should be generated from `section_evidence.json`-level data, not from a second custom traversal. That keeps machine and human outputs aligned.

## Data Model Notes

The first version should remain section-level only.

Recommended supporting concepts:

- `SectionEvidence`
- `SectionEvidenceSource`
- internal citation-id collection per section

The final output should not require downstream consumers to understand raw HTML anchors.

## Error Handling

This phase should remain tolerant.

- If citation anchors are missing or cannot be resolved, the article conversion should still succeed.
- Missing mappings should result in empty section evidence rather than a hard failure.
- The failure mode should be incomplete evidence, not failed article export.

## Testing Strategy

The first implementation should add:

1. normalization tests
   - citation ids are captured before prose cleanup removes visible markers
   - sections aggregate the correct reference ids

2. writer/render tests
   - `section_evidence.json` matches the expected schema
   - `sources.md` groups sources by section correctly

3. service/output tests
   - bundle writing includes the two new artifacts

4. real-page regression
   - at least one citation-dense person page is validated through fixture or live smoke

## Success Criteria

This phase is successful if:

- `article.md` remains clean and unchanged in style
- each article bundle now includes `section_evidence.json` and `sources.md`
- sections with explicit citations expose the correct reference ids and primary URLs
- sections without explicit citations remain valid and empty rather than breaking the export
- existing output artifacts continue to work without format regressions

## Deferred Follow-Ups

Out of scope for v0.3 phase A:

- paragraph-level evidence mapping
- chunk exports
- source quality tiering
- fallback provenance inference for uncited sections
